phase 2 walkthrough

mbx — read path on Gmail, end to end

A to Z of phase 2: what gets built, in what order, why that order, and what each piece looks like in code. Pin a comment on any section.

01

Goal & scope

phase 2

Get one provider working end-to-end before duplicating the work. Gmail first because OAuth is already wired in phase 1 and the SDK is friendlier than raw IMAP.

By the end of phase 2 the following commands work, against a real Gmail account, with no cache, no fanout, no threading, no mutations:

  • mbx envelope list — paginated envelope listing with filters
  • mbx envelope search — same but with a positional keyword arg
  • mbx message read — full body fetch + MIME parse
  • mbx message export — raw RFC 5322 to stdout
  • mbx folder list — labels surfaced as folders
  • mbx attachment list & mbx attachment download

Everything is single-account (-a <name>), live (no cache reads), read-only. Mutations are phase 4. IMAP is phase 3 — by mirroring the same narrow interfaces phase 2 defines.

Why Gmail first Two phase-1 commits already proved the OAuth round-trip (account auth, account doctor). Phase 2 reuses that token machinery as-is — no new auth surface to design. IMAP needs XOAUTH2 SASL plus optional STARTTLS and capability probing, which is its own debugging hole. Punt to phase 3.
02

Build order & why

Bottom-up by layer, narrow-then-wide by domain. Each step compiles and is testable before the next begins.

  1. Provider skeleton — auth-aware HTTP client wrapping gmail/v1. Nothing domain-aware yet; can users.getProfile as a smoke test.
  2. Envelope domain — EnvelopeLister interface in internal/envelope/, Gmail impl in internal/provider/gmail/, then the envelope list handler.
  3. Envelope search — same interface (search is a flag-shaped variant of list), so this is mostly a handler + small query builder additions.
  4. Message domain — MessageReader interface, Gmail impl, MIME parse + body extraction. Ship message read, then message export (which is just the raw bytes, no parse).
  5. Folder domain — FolderLister, Gmail label-to-folder mapping, folder list handler.
  6. Attachment domain — AttachmentLister + AttachmentDownloader, Gmail impls, the two handlers.

The order is load-bearing in two ways:

  • Each domain interface is born during the first command that needs it. We never define a "future" method speculatively. (See section 04.)
  • Each handler only lands once its domain function exists. The handler can be 25 lines of glue because the domain layer is a ready-made function.
Don't Don't skip ahead and write a kitchen-sink type Provider interface { ... } covering everything phase 2 needs. That's the Java trap AGENTS.md §6 calls out. Many small interfaces, defined by the consumer, satisfied by the provider — see section 04.
03

internal/provider/gmail — auth-aware client

A thin wrapper around google.golang.org/api/gmail/v1 that knows how to refresh OAuth tokens via the existing internal/secret resolver. No domain methods yet — just construction + a smoke probe.

// internal/provider/gmail/client.go
package gmail

import (
    "context"
    "fmt"
    "net/http"

    "golang.org/x/oauth2"
    "google.golang.org/api/gmail/v1"
    "google.golang.org/api/option"

    "github.com/oflabs44/mbx/internal/account"
    "github.com/oflabs44/mbx/internal/secret"
)

// Client wraps the gmail.Service plus the account it was built for.
// Account is on the Client so impl methods can stamp mbx IDs without
// being passed the name at every call site.
type Client struct {
    Account string
    svc     *gmail.Service
}

func New(ctx context.Context, acc account.Account, sec *secret.Resolver) (*Client, error) {
    src, err := acc.OAuthTokenSource(ctx, sec)
    if err != nil {
        return nil, fmt.Errorf("gmail: token source for %q: %w", acc.Name, err)
    }
    httpClient := oauth2.NewClient(ctx, src)
    svc, err := gmail.NewService(ctx, option.WithHTTPClient(httpClient))
    if err != nil {
        return nil, fmt.Errorf("gmail: new service: %w", err)
    }
    return &Client{Account: acc.Name, svc: svc}, nil
}

Two design notes:

  • OAuthTokenSource wraps the cmd read + write_cmd rotation already built in phase 1 (internal/account/auth/). The oauth2.TokenSource contract is what makes this clean — when a refresh fires, the source's Token() calls back into the rotation path, which writes the new refresh token via write_cmd automatically. No new persistence wiring in phase 2.
  • The Gmail SDK's UserId is always the literal string "me". Don't take an email address parameter — confuses the API and confuses the code.
Smoke test Before writing any domain method, land users.getProfile("me") behind a doctor check or a one-off test. Catches "OAuth scope is wrong" or "the token isn't actually refreshing" before they're tangled with envelope-fetching bugs.
04

Interfaces live with the consumer

The shape every domain package follows. Read this once; the rest of phase 2 is repetition.

Each domain function in internal/<domain>/ declares the narrow interface it needs. The provider package satisfies it by writing matching methods — but never imports the interface.

// internal/envelope/list.go
package envelope

import "context"

// Lister is the only capability List requires from a backend.
// Defined here, next to its sole consumer; not in internal/provider.
type Lister interface {
    ListEnvelopes(ctx context.Context, q ListQuery) (page Page, err error)
}

type ListQuery struct {
    Folder        string    // empty = backend default (INBOX)
    Limit         int       // 1..500; default 20
    Cursor        string    // opaque, from previous Page.NextCursor
    Unread        *bool
    Starred       *bool
    HasAttachment *bool
    From          string
    To            string
    After         time.Time
    Before        time.Time
    RawQuery      string    // --query escape hatch
}

type Page struct {
    Envelopes  []Envelope
    NextCursor string
}

func List(ctx context.Context, b Lister, q ListQuery) (Page, error) {
    if q.Limit == 0 {
        q.Limit = 20
    }
    return b.ListEnvelopes(ctx, q)
}
// internal/provider/gmail/envelope.go
package gmail

import (
    "context"

    "github.com/oflabs44/mbx/internal/envelope"
    "google.golang.org/api/gmail/v1"
)

// ListEnvelopes satisfies envelope.Lister. The interface name lives
// in internal/envelope; this package never imports it for the type
// declaration — only for the parameter / return types.
func (c *Client) ListEnvelopes(ctx context.Context, q envelope.ListQuery) (envelope.Page, error) {
    // ... see section 05
}

Why this matters in phase 2 specifically:

  • When phase 3 adds the IMAP backend, internal/provider/imap/ writes its own ListEnvelopes with the same signature. The domain layer doesn't change. That's the validation that the abstraction is actually right.
  • An IMAP server without THREAD capability simply won't satisfy envelope.ThreadSearcher in phase 5. The branch is if t, ok := backend.(envelope.ThreadSearcher); ok — no flag, no probe surface area in the interface.
05

mbx envelope list — Gmail impl

List = users.messages.list for IDs, then a batched users.messages.get per ID with format=metadata for the envelope fields. Two RPCs, not one per message.

func (c *Client) ListEnvelopes(ctx context.Context, q envelope.ListQuery) (envelope.Page, error) {
    call := c.svc.Users.Messages.List("me").
        MaxResults(int64(q.Limit)).
        LabelIds(folderToLabel(q.Folder))
    if q.Cursor != "" {
        call = call.PageToken(q.Cursor)
    }
    if rq := buildGmailQuery(q); rq != "" {
        call = call.Q(rq)
    }
    resp, err := call.Context(ctx).Do()
    if err != nil {
        return envelope.Page{}, mapGmailErr(err)
    }
    if len(resp.Messages) == 0 {
        return envelope.Page{NextCursor: resp.NextPageToken}, nil
    }
    metas, err := c.batchGetMetadata(ctx, resp.Messages)
    if err != nil {
        return envelope.Page{}, err
    }
    out := make([]envelope.Envelope, 0, len(metas))
    for _, m := range metas {
        out = append(out, c.toEnvelope(m))
    }
    return envelope.Page{Envelopes: out, NextCursor: resp.NextPageToken}, nil
}

Three things worth dwelling on:

  • buildGmailQuery assembles q.From, q.Unread, etc. into Gmail's query syntax (from:cfo@x.com is:unread). q.RawQuery is appended verbatim — that's the --query escape hatch from commands.md.
  • The cursor is an opaque string. Right now it's literally Gmail's nextPageToken. That's a tiny leak we should think about — when IMAP arrives, its cursor will be different shape (UID-based). Either commit to "cursor is opaque, provider-defined, never round-tripped between providers" (cleanest), or wrap it in a small base64 envelope tagged with the provider. See section 11.
  • folderToLabel is the inverse of section 09's label→folder mapping. Pull it out so both directions live in one file (internal/provider/gmail/folder.go).
Envelope vs message format=metadata with metadataHeaders=From,To,Subject,Date is the envelope-vs-message split as a wire-format choice. Never call format=full from the envelope path. If you do, every list call pulls every body — that's the perf bug AGENTS.md §3 warns about.
06

mbx envelope search — same lister, different query builder

Search is list with a positional keyword. The Gmail backend doesn't need a new method — the keyword folds into the same Q() string.

// internal/envelope/search.go
type SearchQuery struct {
    ListQuery
    Keywords string // positional ""
}

func Search(ctx context.Context, b Lister, q SearchQuery) (Page, error) {
    lq := q.ListQuery
    // Keywords get OR'd into RawQuery so backend builders only see one place.
    if q.Keywords != "" {
        lq.RawQuery = strings.TrimSpace(lq.RawQuery + " " + q.Keywords)
    }
    return List(ctx, b, lq)
}

This is the payoff of the narrow-interface design: search is a 10-line domain function, not a new backend method. IMAP's search will be different — it'll need to translate keywords into SEARCH BODY / SEARCH TEXT criteria — but that fits in IMAP's buildImapSearch, not in the envelope domain.

Cross-folder by default envelope list defaults to --folder INBOX. envelope search defaults to no folder filter — search is cross-folder per commands.md. Make sure buildGmailQuery only adds label: when q.Folder != "".

internal/message — read & body extraction

Same interface-with-consumer pattern. MessageReader in internal/message/, Gmail impl calls users.messages.get with format=full, then mbx parses the MIME tree itself.

// internal/message/read.go
type Reader interface {
    ReadMessage(ctx context.Context, id mbxid.ID) (RawMessage, error)
}

// RawMessage is the unparsed RFC 5322 bytes plus the immutable
// fields we already have from the protocol (ID, server-side flags).
// Parsing happens in internal/message/parse.go, not in the backend.
type RawMessage struct {
    ID          mbxid.ID
    ThreadID    mbxid.ID
    Flags       []string
    Folders     []string
    Provider    string
    Raw         []byte // RFC 5322
}

type ReadOptions struct {
    PreferHTML  bool
    IncludeRaw  bool
    ExtraHeader []string
    OmitHeaders bool
    MarkSeen    bool // true unless --preview
}

func Read(ctx context.Context, r Reader, id mbxid.ID, opt ReadOptions) (Message, error) {
    raw, err := r.ReadMessage(ctx, id)
    if err != nil {
        return Message{}, err
    }
    return parse(raw, opt)
}

Why parse in the domain layer and not in the backend?

  • The MIME tree is the same shape regardless of provider. Phase 3's IMAP backend will return RawMessage.Raw from FETCH BODY[]; the parse step doesn't change.
  • Body-source selection (plain vs html) is a policy decision, not a protocol detail. Keeping it in internal/message/parse.go means there's one place to fix if the policy changes.

Body-source policy

Plain-first: walk the MIME tree, prefer the first text/plain alternative, fall back to text/html stripped to text via a small html-to-text routine. Mark the chosen path in body_source on the JSON output.

// internal/message/body.go
const (
    BodyPlain    = "text/plain"
    BodyHTMLAsTx = "text/html-as-text" // html stripped because no plain part existed
    BodyHTML     = "text/html"         // user passed --html
    BodyRaw      = "raw"               // user passed --raw
)

func chooseBody(parts []mimePart, opt ReadOptions) (text, source string) {
    if opt.IncludeRaw {
        return "", BodyRaw
    }
    if opt.PreferHTML {
        if h := pickType(parts, "text/html"); h != nil {
            return string(h.body), BodyHTML
        }
    }
    if p := pickType(parts, "text/plain"); p != nil {
        return string(p.body), BodyPlain
    }
    if h := pickType(parts, "text/html"); h != nil {
        return htmlToText(h.body), BodyHTMLAsTx
    }
    return "", "" // empty body — valid for some calendar invites etc
}
Decide before writing Open question: what does htmlToText use? Options: write ~80 lines using golang.org/x/net/html tokeniser (preferred — "a little copying"), or pull jaytaylor/html2text (~5KLOC dep, but battle-tested). I lean toward writing it ourselves; the corpus we hit is mostly marketing emails where simple block-element handling is enough. Pin a comment if you disagree — section 11.

mbx message export — raw bytes, no parse

Trivial once RawMessage exists. The handler writes raw.Raw to stdout and exits. No JSON envelope wrapping for this command — it's binary-clean by design (pipeable into msmtp, mu, etc.).

// cmd/mbx/message_export.go
func runMessageExport(cmd *cobra.Command, args []string) error {
    id, err := mbxid.Parse(args[0])
    if err != nil {
        return err
    }
    raw, err := message.RawByID(cmd.Context(), backendFor(id), id)
    if err != nil {
        return err
    }
    _, err = cmd.OutOrStdout().Write(raw.Raw)
    return err
}
Output contract carve-out message export deliberately violates ADR-0004 — it emits raw bytes, not a JSON envelope. That's a documented exception in commands.md, but worth surfacing in internal/output/ as an explicit "raw mode" so a future reviewer doesn't assume it's an oversight.
09

internal/folder — labels surfaced as folders

Gmail labels split two ways in mbx output: some become Folders (multi-valued container), some become Flags (boolean status). The split lives in one place.

// internal/provider/gmail/folder.go
// Labels that map to Flag, not Folder. mbx never surfaces these as folders.
var labelsAsFlag = map[string]string{
    "UNREAD":   "unread",   // mbx Flag vocabulary
    "STARRED":  "flagged",
    "IMPORTANT":"",         // dropped — not in mbx Flag vocab and not a folder
}

// Labels Gmail uses internally that aren't user-meaningful as folders.
var labelsHidden = map[string]bool{
    "CHAT": true, "CATEGORY_PROMOTIONS": true, "CATEGORY_UPDATES": true,
    "CATEGORY_SOCIAL": true, "CATEGORY_FORUMS": true,
}

func (c *Client) ListFolders(ctx context.Context) ([]folder.Folder, error) {
    resp, err := c.svc.Users.Labels.List("me").Context(ctx).Do()
    if err != nil { return nil, mapGmailErr(err) }
    out := make([]folder.Folder, 0, len(resp.Labels))
    for _, l := range resp.Labels {
        if _, isFlag := labelsAsFlag[l.Id]; isFlag { continue }
        if labelsHidden[l.Id] { continue }
        out = append(out, folder.Folder{
            Name:   labelDisplayName(l),  // INBOX → "INBOX", "Label_42" → user-set name
            Count:  l.MessagesTotal,
            Unread: l.MessagesUnread,
        })
    }
    return out, nil
}

The same map drives the envelope-side flagsFromLabels + foldersFromLabels functions. One source of truth for the split — don't re-list flag labels in envelope.go.

Categories Gmail's CATEGORY_* tabs are dropped above. Some users will want them as folders. Decision now: drop them by default; if anyone asks, surface them under folder.aliases.* or a config flag. Keeping the default surface clean matters more than completeness.

Attachments — list + download

Two interfaces, one for cheap metadata, one for the bytes. Same separation as envelope-vs-message: don't pay download cost on a list call.

// internal/attachment/attachment.go
type Lister interface {
    ListAttachments(ctx context.Context, msgID mbxid.ID) ([]Meta, error)
}
type Downloader interface {
    DownloadAttachment(ctx context.Context, attID mbxid.ID) (io.ReadCloser, error)
}

type Meta struct {
    ID       mbxid.ID // gmail:work:<msg>:att-N — encoded by mbxid pkg
    Filename string
    Size     int64
    MIME     string
}

The Gmail backend implements ListAttachments by calling users.messages.get with format=full and walking the MIME parts that have a non-inline disposition. Two phase-2 codepaths now call format=full on the same message: message read and attachment list. That's fine — the cache (phase 6) will collapse it; phase 2 keeps both straightforward and re-fetches.

Download maps to users.messages.attachments.get. The mbx attachment ID encodes gmail:<account>:<msg>:att-N; the att-N suffix is the index into the message's parts array (deterministic given the same MIME tree). Decoding splits and re-fetches just that one part.

Output destination attachment download's -o defaults to the config's attachment_dir. If the config doesn't set one, default to $XDG_DOWNLOAD_DIR if set, else CWD. Don't introduce a custom MBX_ATTACHMENT_DIR env var alone — layer it with XDG and the flag.
11

Open decisions before code lands

decide first

Five small calls that are cheaper to make now than to revisit mid-phase.

D1. Cursor opacity

Two options. Recommended: opaque provider-string, no envelope wrapping. The cursor is documented as not portable across providers (it's not portable across folders either, in IMAP's case). Wrapping it in a base64 tag is over-engineering until we have a use case. Trade-off: a future tool that stores cursors will see Gmail's raw tokens in their JSON.

D2. html-to-text implementation

Recommended: write it in ~80 lines using golang.org/x/net/html tokeniser. Block elements emit \n, <br> emits \n, drop script/style, collapse whitespace. We can always pull a library later if specific email patterns break it. Trade-off: marketing-email rendering won't be pixel-perfect; that's fine for an agent CLI.

D3. Body-encoding edge cases

Quoted-printable and base64 transfer encodings are handled by net/mail's reader for free. Charset conversion (Windows-1252, ISO-8859-1) is not — needs golang.org/x/text/encoding/charmap. Decision: handle UTF-8 + Latin-1 + Windows-1252 explicitly in phase 2; everything else falls back to "decode as UTF-8 with replacement" and warns on stderr at --verbose.

D4. mbx folder list count semantics

Gmail's messagesTotal includes archived; threadsTotal reflects what users see. Recommended: surface both — top-level count = messagesTotal (matches IMAP's count semantics), gmail.threads_total in the provider-extras subobject. Keeps the cross-provider field stable.

D5. Rate-limit / 429 handling

Gmail's API quota is generous but real. Recommended: don't add a retry layer in phase 2. Map 429 to provider.rate_limited (exit 20), let the agent script decide whether to back off. Adding cenkalti/backoff is an easy phase-8 polish if it actually bites.

Error mapping — Gmail SDK to mbx codes

One file in internal/provider/gmail/ translates the Gmail SDK's googleapi.Error into the mbx error codes from ADR-0004. The domain layer never sees a Gmail-typed error.

// internal/provider/gmail/errors.go
func mapGmailErr(err error) error {
    var gerr *googleapi.Error
    if !errors.As(err, &gerr) {
        return fmt.Errorf("gmail: %w", err)
    }
    switch {
    case gerr.Code == 401:
        return output.Errf("auth.refresh_failed", "Gmail rejected the OAuth token")
    case gerr.Code == 403 && strings.Contains(gerr.Message, "quota"):
        return output.Errf("provider.rate_limited", "Gmail quota exceeded")
    case gerr.Code == 404:
        return output.Errf("provider.not_found", "message or label not found")
    case gerr.Code == 429:
        return output.Errf("provider.rate_limited", "Gmail rate limit hit")
    case gerr.Code >= 500:
        return output.Errf("provider.network_timeout", "Gmail upstream error: %s", gerr.Message)
    }
    return fmt.Errorf("gmail: %w", err)
}

The 401 path is interesting: phase 1's token source already auto-refreshes on each call. If the SDK still bubbles a 401 here, the refresh itself failed — that's why auth.refresh_failed is the right code, not auth.invalid_credentials.

What phase 2 unblocks

Mostly: phase 3 becomes pattern replication, not new design.

  • Phase 3 (IMAP) — every domain interface phase 2 defines is the contract phase 3's internal/provider/imap/ implements. The shape of EnvelopeLister, MessageReader, etc., is now load-bearing. If something feels wrong about an interface in phase 2, fix it now — phase 3 will calcify it.
  • Phase 4 (write path) — adds new interfaces (EnvelopeFlagger, MessageMover, ...) but follows the same shape. Phase 2's pattern is the phase-4 template.
  • Phase 5 (threading) — envelope.ThreadSearcher is an opt-in interface alongside EnvelopeLister. The capability-via-type-assertion idiom we set up in phase 2 (sec. 04) is what makes threading a clean optional feature.
  • Phase 6 (cache) — the Envelope and Message structs phase 2 defines become the schema for the SQLite cache. Get the field set right now.
The one thing to over-invest in The Envelope and Message struct shapes. They're the JSON contract per ADR-0004, the cache schema per ADR-0003, and the cross-provider lingua franca. Everything else in phase 2 is reasonably easy to refactor; these shapes aren't.
