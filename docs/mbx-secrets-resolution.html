mbx · internal/secret

how secrets are resolved

A tagged-sum read path, a single write path, and a deliberate rejection of provider lock-in.

why

The two constraints that shape everything

mbx must be both unopinionated about which secret store you use and never persist secrets to disk. Those two together rule out almost every off-the-shelf shape.

The first constraint kills the imbox-style approach of compiling against a specific SDK (in imbox's case, 1Password). If mbx hardwired any one tool, the user couldn't move secrets between stores without a fork. The second constraint kills the himalaya-style fallback of "default to a file" — even an encrypted one. We treat the user's existing secret store (whatever it is) as the source of truth.

The shape that survives both: a tagged sum for reads, declared per-secret in TOML config, and a single shell-command hook for writes, declared once per rotating secret. This is documented in ADR-0001; the rest of this page explains how it actually executes.

read

Three read variants, one TOML knob

Every secret declares exactly one of raw, keyring, or cmd. The variant tells mbx where to fetch the value from at read time.

The Go type is a struct with optional fields plus a method that reports which one is set:

type Secret struct {
    Raw      string `toml:"raw,omitempty"`
    Keyring  string `toml:"keyring,omitempty"`
    Cmd      string `toml:"cmd,omitempty"`
    WriteCmd string `toml:"write_cmd,omitempty"`
}

func (s *Secret) Variant() (SecretVariant, string) {
    switch {
    case s == nil:        return SecretNone, ""
    case s.Raw != "":     return SecretRaw, s.Raw
    case s.Keyring != "": return SecretKeyring, s.Keyring
    case s.Cmd != "":     return SecretCmd, s.Cmd
    default:              return SecretNone, ""
    }
}

In the TOML, that looks like one of:

# raw (testing only — secret on disk)
[accounts.gmail-personal.backend.auth.refresh-token]
raw = "1//0eXXXXXXXX..."

# keyring (OS keychain via zalando/go-keyring)
[accounts.gmail-personal.backend.auth.refresh-token]
keyring = "mbx-gmail-personal-refresh-token"

# cmd (universal escape hatch — anything that prints to stdout)
[accounts.gmail-personal.backend.auth.refresh-token]
cmd = "op read op://Dev/mbx-gmail-personal/refresh-token"

Setting more than one of raw, keyring, cmd on the same secret is rejected at config-load time as ErrAmbiguousSecret — there's no precedence rule to remember.

Himalaya parity The three-variant tagged sum is lifted verbatim from himalaya's config shape (and ADR-0001 makes this an explicit invariant — you can paste a himalaya secret block into mbx config and it parses). The only field name mbx adds is write_cmd, which is the part himalaya doesn't have.
dispatch

How Read() dispatches

A single switch on the variant. No interfaces, no plugin registry — just three branches and a fallthrough error.

func Read(ctx context.Context, s *config.Secret) (string, error) {
    if s == nil {
        return "", ErrNoVariant
    }
    v, val := s.Variant()
    switch v {
    case config.SecretRaw:
        return val, nil
    case config.SecretKeyring:
        pwd, err := keyring.Get(val, keyringUser)
        if err != nil {
            if errors.Is(err, keyring.ErrNotFound) {
                return "", fmt.Errorf("%w (%s)", ErrKeyringNotFound, val)
            }
            return "", fmt.Errorf("%w (%s): %v", ErrKeyringRead, val, err)
        }
        return pwd, nil
    case config.SecretCmd:
        return runCmd(ctx, val, nil)
    default:
        return "", ErrNoVariant
    }
}

What each branch actually does:

  • SecretRaw — return the value inline. No I/O. Useful only for tests; documented as such.
  • SecretKeyring — call zalando/go-keyring.Get(service, user). service is the user's TOML value (e.g. "mbx-gmail-personal-refresh-token"); user is a single fixed constant ("mbx") so the user never has to think about it.
  • SecretCmd — execute the command via the system shell, capture stdout, trim one trailing newline. See the next section.

Two sentinel errors carry the keyring failure modes separately: ErrKeyringNotFound when the item is missing (user remedy: run write_cmd to seed it, or check the spelling) versus ErrKeyringRead for everything else (backend locked, daemon down, etc.). account doctor will use this distinction to suggest the right next step.

exec

runCmd: the universal escape hatch

The cmd variant runs the user's string through the system shell. That means pipes, redirects, environment substitution, anything POSIX sh -c understands — and the user supplies the entire string.

func runCmd(ctx context.Context, cmd string, stdin io.Reader) (string, error) {
    var c *exec.Cmd
    if runtime.GOOS == "windows" {
        c = exec.CommandContext(ctx, "cmd.exe", "/c", cmd)
    } else {
        c = exec.CommandContext(ctx, "sh", "-c", cmd)
    }
    c.Stdin = stdin
    var out, errBuf bytes.Buffer
    c.Stdout = &out
    c.Stderr = &errBuf
    if err := c.Run(); err != nil {
        if ctxErr := ctx.Err(); ctxErr != nil {
            return "", fmt.Errorf("%w: %w", ctxErr, err)
        }
        stderr := strings.TrimSpace(errBuf.String())
        if stderr == "" {
            return "", fmt.Errorf("%w: %v", ErrCmdFailed, err)
        }
        return "", fmt.Errorf("%w: %v: %s", ErrCmdFailed, err, stderr)
    }
    return strings.TrimSuffix(out.String(), "\n"), nil
}

Three deliberate behaviours worth knowing about:

  1. Shell, not exec. mbx uses sh -c "$cmd" on POSIX (and cmd.exe /c on Windows) so the user can write op read op://... or pass show gmail/refresh | head -1 without thinking. This matches what himalaya does.
  2. Context propagates. The ctx.Err() check after c.Run() distinguishes "your --timeout fired" from "the command exited non-zero". Downstream callers can errors.Is(err, context.DeadlineExceeded) and suggest a different remedy.
  3. One trailing newline gets trimmed. Every secret CLI (op, pass, security) ends its output with a newline; if mbx didn't strip it, it'd ship as part of the token and break HTTP headers. We trim exactly one, so a legitimate multi-line secret (e.g. a PEM block) still works.
Security note The user controls the full command string in their config. mbx runs it through sh -c verbatim. This is intentional (it's the same trust model as git config credential.helper) — but it also means anyone who can write your config can run arbitrary code as you. Don't accept config files from untrusted sources.
write

Write goes through one path, always

Reads have three variants. Writes have one: write_cmd. The OS keyring is not a built-in write target — it's just one possible write_cmd destination among many.

This is the part where mbx deliberately diverges from himalaya. Himalaya can write rotated OAuth tokens to the OS keyring (its keyring variant is read+write). That sounds convenient but creates a real problem: if your other secrets live in 1Password, the rotated token silently lands in keychain — different store, no one tells you, you find out months later when you migrate machines and the token is missing.

mbx requires the user to declare a write_cmd for any rotating secret (OAuth refresh-token). The command receives the new value on stdin; its exit code is success or failure.

func Write(ctx context.Context, s *config.Secret, value string) error {
    if s == nil || s.WriteCmd == "" {
        return ErrNoWriteCmd
    }
    _, err := runCmd(ctx, s.WriteCmd, strings.NewReader(value))
    return err
}

The TOML pairs read and write next to each other:

[accounts.gmail-personal.backend.auth.refresh-token]
cmd       = "op read op://Dev/mbx-gmail-personal/refresh-token"
write_cmd = "op item edit mbx-gmail-personal refresh-token[password]=-"

Both go through the same shell. The =- at the end of the op item edit is the convention 1Password uses to mean "read from stdin." Other stores have equivalents (pass insert -m, security add-generic-password, etc.) — the docs will ship recipes for the common ones.

Why not auto-write to keyring when read is keyring? It's tempting. But "infer the write target from the read target" creates an invisible coupling: if the user later moves their tokens to cmd, mbx silently keeps writing to keyring. Explicit write_cmd means there's only ever one place rotated tokens go, and the config declares it.
guard

Where write_cmd becomes mandatory

For OAuth refresh-tokens, mbx refuses to run account auth without a write_cmd configured. There's no fallback. This is the price of "never on disk."

An OAuth flow ends with mbx holding a freshly-issued refresh-token. If mbx had no way to persist it, the user would have to run account auth every session — useless. So either we accept "no write_cmd" and silently invent a place to put the token (back to lock-in), or we make it mandatory up front with a friendly error.

The friendly error matters. mbx account auth gmail-personal without a write_cmd doesn't crash — it errors before opening the browser, points at the missing field, and shows a paste-ready recipe for the user's likely backend (1Password / pass / keyring). This logic isn't implemented yet (it lands with phase 1, account auth), but the design intent is in ADR-0001 and AGENTS.md §4.

Passwords (the password-auth variant for IMAP accounts) are not rotating, so they don't need write_cmd. They're configured once, never overwritten by mbx.

walk

A full example: Gmail via 1Password

What happens, step by step, when you run mbx envelope list -a gmail-personal with secrets in 1Password.

The relevant config block:

[accounts.gmail-personal]
type = "gmail"
email = "you@gmail.com"

[accounts.gmail-personal.backend.auth]
type      = "oauth2"
client-id = "1234.apps.googleusercontent.com"
auth-url  = "https://accounts.google.com/o/oauth2/v2/auth"
token-url = "https://www.googleapis.com/oauth2/v3/token"

[accounts.gmail-personal.backend.auth.client-secret]
cmd = "op read op://Dev/mbx-gmail-personal/client-secret"

[accounts.gmail-personal.backend.auth.refresh-token]
cmd       = "op read op://Dev/mbx-gmail-personal/refresh-token"
write_cmd = "op item edit mbx-gmail-personal refresh-token[password]=-"

The flow at runtime:

  1. internal/config.Load() parses the TOML, validates that client-secret and refresh-token each declare exactly one read variant, and returns the typed *Account.
  2. The Gmail provider (phase 2 — not yet built) needs to call the token endpoint. It calls secret.Read(ctx, account.Backend.Auth.ClientSecret) and secret.Read(ctx, account.Backend.Auth.RefreshToken).
  3. Both Read calls dispatch to SecretCmd. Each runs op read op://... via sh -c, captures stdout, trims one trailing newline. Two short-lived op subprocesses, no on-disk caching.
  4. Google returns a fresh access-token plus (sometimes) a rotated refresh-token. If the refresh-token rotated, the provider calls secret.Write(ctx, account.Backend.Auth.RefreshToken, newToken).
  5. Write sees WriteCmd is set, execs op item edit mbx-gmail-personal refresh-token[password]=-, pipes the new token to stdin. op updates the 1Password item. mbx never wrote a thing to disk.
  6. Provider makes the actual Gmail API call with the access token. Returns envelopes.

Switch to pass tomorrow? Change two lines:

cmd       = "pass show mbx-gmail-personal/refresh-token"
write_cmd = "pass insert -m -f mbx-gmail-personal/refresh-token"

No code change. No mbx update. The whole point.

fail

What failures actually look like

The secret package returns plain Go errors wrapping sentinel values. The cmd/mbx layer translates them to JSON-output error codes; the secret package itself stays unaware of the output contract.

Sentinels:

  • ErrNoVariant — the *config.Secret was nil or had no read variant set. This shouldn't reach runtime because config.Load() rejects it earlier; the sentinel exists for defense-in-depth.
  • ErrCmdFailed — the shell command exited non-zero. The error message includes stderr (trimmed) so the user sees what their tool actually complained about.
  • ErrKeyringRead — keyring backend reachable but the read itself failed for some reason other than "not found."
  • ErrKeyringNotFound — the keyring backend works but the item doesn't exist. Different sentinel because the remedy is different: seed the item via write_cmd, not unlock the keychain.
  • ErrNoWriteCmd — Write was called on a secret that has no write_cmd configured. Caller surface (e.g. account auth) translates this into the actionable "configure a write_cmd" message.

Context cancellation is preserved as a wrapped error, not swallowed: errors.Is(err, context.DeadlineExceeded) works in callers, which means --timeout failures during a slow op read get reported as timeouts rather than as "command failed."

gotchas

Things worth knowing the first time

A handful of small surprises that fall out of the design.

  • Free functions, no Resolver. Early drafts had a Resolver struct with a constructor. Then the simplifier pointed out it was a single-field wrapper with no behavior swap, so it became two top-level functions. The package surface is two functions and a few sentinels. That's it.
  • Flat password-mode fields under OAuth blocks are rejected. If you migrate a password account to OAuth and forget to delete the leftover raw = "...", mbx errors at config-load with ErrUnexpectedSection. This catches a real migration-class footgun that silent-failure-hunter flagged early.
  • Account names must avoid : and %. Because : is the separator in mbx IDs and % is the escape char (see ADR-0002), account names can't use them. The config validator will enforce this in a follow-up commit; until then, just don't.
  • No keyring backend dependency, just the read variant. mbx ships zalando/go-keyring for the read side. If you never configure a keyring = "...", the dep is dead weight at runtime but doesn't change behaviour. We accepted the dep instead of removing the variant because the himalaya-parity argument outweighed the ~200 KB of binary size.
  • Writes don't trim newlines. Write sends the value to stdin exactly as the caller passed it. If your token has a trailing newline because the upstream API returned one, that's what gets stored. We only trim on read, where it's a convention from CLI tools.
